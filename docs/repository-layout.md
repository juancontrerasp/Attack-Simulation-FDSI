# Repository Layout

This repository mixes source code, documentation, research outputs, and demo artifacts. The goal of this page is to separate what is canonical from what is generated so the workspace stays understandable without breaking the existing scripts.

## Canonical Source

| Path | Purpose |
|---|---|
| `attack-engine/` | Java attack simulator and attack classes |
| `stride-agent/` | STRIDE analyzer, parsers, providers, cache, and comparison tools |
| `reference-architecture/` | Secure benchmark application, architecture docs, and comparison helpers |
| `scripts/` | Report consolidation and helper scripts |
| `threat-models/` | Manual baseline analyses used for comparison |

## Documentation

| Path | Purpose |
|---|---|
| `README.md` | Main landing page and workflow overview |
| `docs/` | Schemas, threat mappings, controls catalog, metric formulas, and layout notes |
| `DEMO_5MIN_WINDOWS.md` | Demo script for Windows / PowerShell |

## Runtime Inputs Kept at the Root

These files are part of the runtime contract and are expected by current scripts.

| File | Why it stays at the root |
|---|---|
| `package.json` | Entry point for npm scripts |
| `launch_attack.ps1` / `run-full-analysis.ps1` | Windows-native execution entrypoints |
| `launch_attack.sh` / `run-full-analysis.sh` | Unix/macOS compatibility entrypoints |
| `passwords.txt` | Attack input wordlist |

| File | Produced by | Status |
|---|---|---|
| `threats-output.json` | `npm run stride:analyze` / `stride-agent/index.js` | Generated |
| `results.json` | `launch_attack.ps1` / Java attack engine | Generated |
| `combined-report.json` | `scripts/combine-results.js` | Generated |
| `metrics-report.json` | `compare.js` | Generated |
| `metrics-report.html` | `compare.js` | Generated |

| `dashboard-standalone.html` | `create-standalone-dashboard.py` | Generated |
| `design-feedback.*` | Design feedback generation | Generated |

## Organization Rules

- Canonical inputs live in source or docs folders.
- Generated outputs stay out of version control when possible and are covered by `.gitignore`.
- Root-level generated files are tolerated only when the current pipeline requires stable relative paths.
- If a report is reusable as documentation, the canonical copy belongs under `docs/` or `reference-architecture/`, not as an ad hoc root artifact.
